# Kernel Type Glossary

This glossary is derived from the NVTX ranges currently added in `cs336_systems/benchmark.py` and `cs336-basics/cs336_basics/model.py`. Use it to map Nsight timeline ranges back to the major modules of the Transformer benchmark and model implementation.

| NVTX Range | Core Module | Common Kernel Type / Compute Pattern | What It Means |
| --- | --- | --- | --- |
| `benchmark.warmup` | Warmup phase | mixed stage | Pre-measurement steps used to stabilize execution before timing. |
| `benchmark.measure` | Measured phase | mixed stage | The main timed region that should be used for performance reporting. |
| `benchmark.forward` | Forward pass | mixed stage | End-to-end forward path including embedding, attention, FFN, normalization, and LM head. |
| `benchmark.loss` | Loss computation | reduction / elementwise | Cross-entropy and related reduction work after logits are produced. |
| `benchmark.backward` | Backward pass | backward kernels | Gradient computation for the full forward graph. |
| `benchmark.zero_grad` | Gradient reset | memory / elementwise | Clears or resets gradient buffers before a training step. |
| `benchmark.optimizer_step` | Optimizer update | optimizer update / elementwise | Parameter update stage that appears in `train-step` mode. |
| `model.embedding` | Token embedding | gather / embedding lookup | Maps token ids to dense embeddings. |
| `model.layer.` | One Transformer block | mixed stage | The full per-layer region, which expands into attention and FFN work. |
| `block.attention` | Self-attention sublayer | attention-related kernels | The attention branch inside a Transformer block. |
| `block.ffn` | Feed-forward sublayer | GEMM + activation + elementwise | The FFN or SwiGLU branch inside a Transformer block. |
| `model.ln_final` | Final LayerNorm | normalization / reduction | Final normalization before vocabulary projection. |
| `model.lm_head` | Vocabulary projection | GEMM | Projects hidden states to vocabulary logits. |
| `attention.qkv_proj` | Q/K/V projections | GEMM | Linear projections that produce query, key, and value tensors. |
| `attention.rearrange_qkv` | QKV head rearrangement | reshape / transpose / memory movement | Splits hidden dimension into attention heads. |
| `attention.rope` | Rotary position encoding | elementwise | Applies RoPE to Q and K. |
| `attention.causal_mask` | Causal mask construction | compare / mask generation | Builds the autoregressive mask for attention. |
| `attention.core` | Core attention region | mixed attention kernels | Wrapper around the main scaled dot-product attention computation. |
| `attention.scores` | Attention score computation | GEMM / batched matmul | Computes `Q @ K^T / sqrt(d_k)`. |
| `attention.mask` | Attention masking | elementwise | Applies the mask, typically writing invalid positions to `-inf`. |
| `attention.softmax` | Attention normalization | softmax / reduction | Normalizes attention scores over the key dimension. |
| `attention.value_matmul` | Weighted value aggregation | GEMM / batched matmul | Computes `softmax(QK^T) @ V`. |
| `attention.concat_heads` | Head concatenation | reshape / transpose / memory movement | Reassembles multi-head outputs into hidden dimension layout. |
| `attention.output_proj` | Attention output projection | GEMM | Final linear projection after attention head aggregation. |

Recommended reading order in Nsight:

1. Start from `benchmark.measure` to isolate the timed region.
2. Then compare `benchmark.forward`, `benchmark.backward`, and `benchmark.optimizer_step` to understand stage-level cost.
3. Expand `block.attention` and `attention.core` when the hotspot is in self-attention.
4. Compare `attention.scores`, `attention.softmax`, and `attention.value_matmul` when analyzing attention-internal kernel balance.

Using the NVTX ranges in the forward-pass sqlite traces, I compared `attention.softmax` against two matmul views: core attention matmuls (`attention.scores` + `attention.value_matmul`) and all attention matmuls (`attention.qkv_proj` + `attention.scores` + `attention.value_matmul` + `attention.output_proj`).

| Model | Softmax per forward (ms) | Core attention matmuls per forward (ms) | Core matmul / softmax | All attention matmuls per forward (ms) | All matmul / softmax |
| --- | ---: | ---: | ---: | ---: | ---: |
| small | 3.768 | 5.556 | 1.474x | 17.266 | 4.582x |

Answer: softmax is clearly slower than its FLOP count alone would suggest. Even though the matrix multiplications have many more FLOPs, the measured runtime gap is only about 1.3-1.5x for the core attention matmuls, or about 4.3-4.6x if we include QKV and output projections; this is much smaller than the FLOP gap, which is consistent with softmax being more memory-bound and reduction-heavy while GEMMs are much more compute-efficient.

* 我的实验现象基本是合理的：softmax 通常比 GEMM 更难充分利用 GPU，因为它算术强度低、需要多轮访存并包含归约操作，因此更偏 memory-bound。随着 seq\_len 增大，虽然 softmax 和 attention matmul 的 FLOPs 都按 O(seq2)O(seq^2)**O**(**se**q**2**) 增长，但 softmax 的访存和归约开销更容易成为瓶颈，而 GEMM 往往还能保持较高计算效率，所以 softmax 相对会变得更慢。

# Softmax vs Matmul：简化模型下的计算量与计算强度

## 设定

为了简化比较，假设：

- softmax 和 matmul 读取数据时都已经优化得比较好
- 数据已放入 shared memory
- 不考虑反复从 global memory 回读
- softmax 只考虑以下步骤：
  1. 减去最大值
  2. 做 exp
  3. 对 exp 结果求和
  4. 除以这个和完成归一化
- matmul 按普通乘加计算

---

## 1. Softmax 小例子

设一行输入长度为 4：

`x = [1, 2, 3, 4]`

### 计算步骤

1. **减最大值**
   
   - 4 次减法
2. **做 exp**
   
   - 4 次 exp
3. **exp 后求和**
   
   - 3 次加法
4. **归一化**
   
   - 4 次除法

### 总计算次数

- 4 次减法
- 4 次 exp
- 3 次加法
- 4 次除法

总共约：

`15 次操作`

### 数据量

如果是 `float32`：

- 输入 4 个数：16 bytes
- 输出 4 个数：16 bytes

总数据量：

`32 bytes`

### 计算强度

`AI_softmax ≈ 15 / 32 ≈ 0.47`

---

## 2. Matmul 小例子

设：

`A(1x4) × B(4x4) = C(1x4)`

### 计算量

每个输出元素需要：

- 4 次乘法
- 3 次加法

共有 4 个输出元素，所以总共：

- 16 次乘法
- 12 次加法

按 GEMM 常见口径：

`FLOPs = 2 × 1 × 4 × 4 = 32`

### 数据量

- 输入向量 A：4 个 float = 16 bytes
- 输入矩阵 B：16 个 float = 64 bytes
- 输出 C：4 个 float = 16 bytes

总数据量：

`96 bytes`

### 计算强度

`AI_matmul ≈ 32 / 96 ≈ 0.33`

### 说明

这个结果看起来比 softmax 还低，但这是因为例子太小，matmul 的数据复用优势还没有充分体现出来。

---

## 3. 更合理的 Matmul 例子：4×4 矩阵乘法

设：

`A(4x4) × B(4x4) = C(4x4)`

### FLOPs

`2 × 4 × 4 × 4 = 128`

### 数据量

- A：16 个 float = 64 bytes
- B：16 个 float = 64 bytes
- C：16 个 float = 64 bytes

总数据量：

`192 bytes`

### 计算强度

`AI_matmul ≈ 128 / 192 ≈ 0.67`

这时已经高于 softmax 的约 0.47。

---

## 4. 推广结论

### Softmax（长度为 n）

#### 操作数

- 减最大值：`n`
- exp：`n`
- 求和：`n - 1`
- 除法：`n`

总共约：

`4n`

#### 数据量

- 输入 `n` 个数
- 输出 `n` 个数

总共约：

`2n 个 float = 8n bytes`

#### 计算强度

`AI_softmax ≈ 4n / 8n = 0.5`

**结论：softmax 的计算强度大致是常数级。**

---

### Matmul（n×n）

设：

`A(nxn) × B(nxn) = C(nxn)`

#### FLOPs

`2n^3`

#### 数据量

- A：`n^2`
- B：`n^2`
- C：`n^2`

总共：

`3n^2 个 float = 12n^2 bytes`

#### 计算强度

`AI_matmul ≈ 2n^3 / 12n^2 = n / 6`

**结论：matmul 的计算强度会随着矩阵规模 `n` 线性增长。**

---

## 5. 最终结论

- 在这个理想化模型下，**softmax 的计算强度大约是常数级**
- **matmul 的计算强度会随着规模增大而增加**
- 因此：
  - softmax 更容易成为 **memory-bound / reduction-bound**
  - matmul 更容易高效利用 GPU，尤其在大规模情况下更明显

一句话总结：

> 即使假设 softmax 和 matmul 都能很好地利用 shared memory，softmax 的每个元素参与的计算仍然较少，因此计算强度基本保持常数；而 matmul 中一个元素会参与大量乘加运算，所以计算强度会随问题规模上升，这也是 matmul 更容易充分利用 GPU 的根本原因。
